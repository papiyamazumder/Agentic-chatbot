🤖 KPMG PMO AI Chatbot

Multi-Agent RAG Chatbot — Full Project Documentation for AI Role Interviews

What is this project?

A multi-agent AI chatbot built for KPMG's Project Management Office (PMO). Instead of one monolithic bot, it uses 4 specialized AI agents — each expert in a different domain — orchestrated by a LangGraph router that decides which agent should handle each query.

Stack: Python FastAPI LangGraph Groq LLaMA 3.1 FAISS Streamlit

Why multi-agent instead of one agent?

Definition: Multi-agent architecture splits responsibilities across specialized agents. Each agent has its own tools, prompts, and data sources — making the system modular, testable, and extensible.

A single agent would need to handle document Q&A, SQL queries, ticket creation, email sending, and log analysis — all in one prompt. That leads to confusion, hallucination, and poor routing. With 4 agents, each one does ONE thing very well.

🧭 3-Tier Hybrid Router

Definition: The router is the "brain" that decides which agent handles a query. Instead of one approach, we use 3 tiers to balance speed, cost, and accuracy.

Tier 1 — Keyword Matching

How: Scan query for trigger words from 4 keyword lists (~15 words each).

Speed: <1ms | Cost: Free | Handles: ~70% of queries

Example: "Raise a ticket for login issue" → keyword "ticket" found → Helpdesk Agent

Priority order: Helpdesk > Workflow > API > Retrieval (checked in this order)

Use case: Fast, deterministic routing for obvious queries that contain clear intent words.

Tier 2 — Embedding Cosine Similarity

How: Embed query with all-MiniLM-L6-v2 (384-dim vector). Compare against pre-embedded "anchor sentences" per agent using cosine similarity. Route to highest-scoring agent if score ≥ 0.35.

Speed: ~5ms | Cost: Free (local model) | Handles: ~20% of queries

Example: "I can't connect to the company network from home" → no keywords match → embed → helpdesk anchors score 0.502 → Helpdesk Agent

Definition — Cosine Similarity: Measures the angle between two vectors. 1.0 = identical direction, 0 = perpendicular. Used because it's magnitude-invariant — a short query and long anchor can still match well.
Use case: Handles paraphrased queries where the user doesn't use any expected keywords.
Parameter Value Why
threshold 0.35 Below this, similarity is too weak — escalate to Tier 3
anchors per agent 6-8 sentences Representative queries that define each agent's domain
embedding dim 384 Output dimension of all-MiniLM-L6-v2

Tier 3 — Groq LLM Classifier

How: Send the query to Groq (llama-3.1-8b-instant) with a classification prompt. LLM returns one word: retrieval/api/helpdesk/workflow.

Speed: ~300ms | Cost: 1 API call (free tier) | Handles: ~10% of queries

Example: "Can you help me think about our delivery approach?" → keywords ❌ → embeddings all <0.35 ❌ → LLM says "retrieval" ✅

Use case: Definitive fallback for truly ambiguous queries that neither keywords nor embeddings can classify.
Parameter Value Why
temperature 0 Deterministic — always pick the most likely classification
max_tokens 10 Only need one word back
model llama-3.1-8b-instant Fastest Groq model, sufficient for classification

🤖 4 AI Agents — Deep Dive

🔍Retrieval Agent (RAG)

Definition — RAG (Retrieval-Augmented Generation): Instead of relying on the LLM's training data, we RETRIEVE relevant document chunks first, inject them as context, then GENERATE an answer grounded in those chunks. This prevents hallucination.

Pipeline Flow

Query: "What does the SOP say about vendor onboarding?" ↓ Step 1: Embed query → 384-dim vector (all-MiniLM-L6-v2) ↓ Step 2: FAISS search main index → top 5 chunks (L2 distance) Step 3: FAISS search upload index → top 3 chunks (if session has uploads) ↓ Step 4: Merge results (uploads first) → top 7 chunks as context ↓ Step 5: Build prompt: system="Answer ONLY from context" + user=context+query ↓ Step 6: Groq LLM (temp=0.2) → grounded answer citing source documents ↓ Output: "According to the Vendor Onboarding SOP (page 3)..."

Parameters

Parameter Value Purpose
top_k (main) 5 Number of chunks from pre-ingested docs
top_k (upload) 3 Number of chunks from uploaded docs
temperature 0.2 Low = factual, minimal creativity
max_tokens 1024 Enough for detailed answers
chunk_size 500 chars Balance: too small = no context, too large = noise
overlap 100 chars Prevents splitting sentences at chunk boundaries
Use cases: "Summarize the Q3 report", "What's the policy on remote work?", "Find risks in the delivery document"

📥 Ingestion Pipeline

Definition: The ingestion pipeline converts raw documents into searchable vectors. It runs ONCE before the app starts (via python run_ingestion.py).
data/raw_docs/SOP.pdf, Report.docx, KPIs.xlsx ↓ Step 1: pdf_loader.py — Parse each file format to plain text PDF → PyMuPDF (fitz) | DOCX → python-docx | XLSX → pandas ↓ Step 2: chunker.py — Split into 500-char chunks with 100-char overlap "The vendor onboarding process requires..." → chunk_0, chunk_1, ... ↓ Step 3: embedder.py — Encode each chunk → 384-dim float vector all-MiniLM-L6-v2 (HuggingFace, runs locally, free) ↓ Step 4: vector_store.py — Save to FAISS IndexFlatL2 data/vector_store/index.faiss + metadata.pkl
Component File Key Concept
Document Parser pdf_loader.py PyMuPDF for PDF, python-docx for Word, pandas for Excel/CSV
Chunker chunker.py Overlapping windows prevent splitting mid-sentence
Embedder embedder.py Singleton pattern — model loaded once, reused across requests
Vector Store vector_store.py FAISS IndexFlatL2 — brute-force L2 distance, exact results

📤 File Upload Feature

Definition — Session-scoped Storage: Each browser tab gets a unique session_id (UUID). Uploaded docs are indexed into a SEPARATE temp FAISS index under /tmp/kpmg_uploads/{session_id}/. When the user clears chat, the temp files and index are deleted.
User drops "Q4_Report.pdf" into sidebar uploader ↓ Frontend sends POST /upload (multipart form: file + session_id) ↓ Backend: upload_handler.py 1. Save file → /tmp/kpmg_uploads/abc123.../Q4_Report.pdf 2. Parse → pdf_loader._load_pdf() 3. Chunk → 500-char chunks with source "📎 Q4_Report.pdf (uploaded)" 4. Embed → all-MiniLM-L6-v2 5. Add to session FAISS index (separate from main) ↓ User asks: "What does the Q4 report say about risks?" ↓ Retrieval Agent searches BOTH: Main index (pre-ingested docs) → top 5 Session index (uploaded docs) → top 3 Merge → LLM answers → "According to 📎 Q4_Report.pdf (uploaded)..." ↓ "Clear Chat" → deletes /tmp files + in-memory index → fresh session

🧠 Technical Concepts Explained

RAG (Retrieval-Augmented Generation)

What: Instead of asking the LLM from its training data (which can hallucinate), we first RETRIEVE relevant documents, then let the LLM GENERATE an answer grounded in those docs.

Where: Retrieval Agent — every document Q&A query.

Why: Prevents hallucination. The LLM can only cite what we give it.

Vector Embeddings

What: Text converted to a list of 384 floating-point numbers (a "vector") that captures semantic meaning. Similar texts → similar vectors.

Where: Ingestion pipeline (chunks → vectors), Router Tier 2 (query → vector), Upload handler.

Model: all-MiniLM-L6-v2 — 22M params, runs locally on CPU, free, no API needed.

FAISS (Facebook AI Similarity Search)

What: A library for fast nearest-neighbor search on vector data. We use IndexFlatL2 (brute-force L2 distance — exact results, good for <100K vectors).

Where: vector_store.py (main index) + upload_handler.py (session temp index).

Why not a cloud DB? Zero cost, zero latency, zero setup. Swappable to Azure AI Search later.

LangGraph StateGraph

What: A directed graph where nodes are functions and edges define flow. State (a TypedDict) flows through the graph. Conditional entry point routes to one of 4 nodes.

Why not LangChain agents? LangChain ReAct agents use unpredictable tool-calling loops. LangGraph is deterministic — we control exactly which node runs.

Prompt Engineering Techniques Used

Technique Agent Example
System prompt constraints Retrieval "Answer ONLY from the context. If not found, say so."
JSON extraction Helpdesk/Workflow "Return ONLY valid JSON. No explanation."
One-word classification Router Tier 3 "Reply with ONLY one word: retrieval, api, helpdesk, or workflow"
Temperature control All 0 for extraction, 0.2 for factual, 0.3 for formatting
Role-based prompts All "You are an expert PMO AI Assistant for KPMG"

OAuth2 Client Credentials Flow

What: App-only authentication (no user login). App sends client_id + client_secret to Azure AD → gets an access token → uses it to call Graph API.

Where: Outlook (send email), SharePoint (read docs/lists), Teams (post cards).

Flow: POST to login.microsoftonline.com/{tenant}/oauth2/v2.0/token → Bearer token → API headers.

🎯 Interview Questions & Answers

Q: Explain your project in 2 minutes.

"We built a multi-agent RAG chatbot for KPMG's PMO team. It uses LangGraph to orchestrate 4 specialized agents: one for document Q&A using FAISS vector search, one for KPI data from SQL/Jira, one for IT helpdesk tickets via ServiceNow, and one for workflow automation — emails, approvals, RAID logs. The router uses a 3-tier strategy: keyword matching for speed, embedding cosine similarity for semantic understanding, and an LLM fallback for ambiguous queries. Everything runs locally with mock data — only a free Groq API key is needed."

Q: Why multi-agent instead of one agent?

"Separation of concerns. Each agent has its own tools, prompts, and data sources. The retrieval agent uses RAG with FAISS. The API agent queries SQL. The helpdesk agent does CRUD on ServiceNow. Mixing all of this into one prompt would cause confusion and hallucination. Multi-agent also means we can test, debug, and scale each agent independently."

Q: How do you handle hallucination?

"Three safeguards: (1) Grounded generation — the retrieval agent can only use retrieved chunks as context, not its training data. (2) System prompt constraint: 'If the answer is not in the context, say so clearly.' (3) Low temperature (0.2) for factual answers, which reduces creative but inaccurate output."

Q: Why 3 tiers for routing?

"Cost and speed optimization. Keywords handle 70% of queries in under 1ms for free. Embeddings catch paraphrased queries using the local model. The LLM fallback is only used for the ~10% truly ambiguous cases — saving API calls and latency. It's a cascade: cheap and fast first, expensive and accurate last."

Q: What is cosine similarity and why use it?

"It measures the angle between two vectors — 1.0 means identical direction, 0 means no relationship. We use it in Tier 2 routing to compare the query embedding against pre-embedded anchor sentences for each agent. We chose cosine over L2 distance because it's magnitude-invariant: a short query and a long anchor can still match if they're semantically similar."

Q: What is FAISS and why not use a cloud vector database?

"FAISS is Facebook's library for fast similarity search. We use IndexFlatL2, which does exact brute-force search. For our scale (~few thousand chunks), it's faster than any cloud DB, costs nothing, and needs zero setup. The architecture is designed so we can swap to Azure AI Search later by only changing vector_store.py — no agent code changes needed."

Q: How does file upload work?

"Each browser tab gets a UUID session_id. When a user uploads a file, it's saved to /tmp, parsed (PyMuPDF for PDF, python-docx for Word, pandas for Excel), chunked into 500-char pieces, embedded with all-MiniLM-L6-v2, and stored in a session-scoped FAISS index — separate from the main index. The retrieval agent searches both indexes and merges results. On 'Clear Chat', the temp files and index are deleted. This means uploaded docs are never persisted — they exist only for the session."

Q: What would you change for production?

"Three things: (1) Replace FAISS with Azure AI Search for scalability and persistence. (2) Add Azure AD JWT authentication to protect the endpoints. (3) Set up scheduled re-ingestion via Azure Functions timer trigger to keep the docs index fresh. The architecture already supports all of this — it's just config changes, no agent rewrites."
