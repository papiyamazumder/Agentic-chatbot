langgraph
Multi-agent orchestration
P1 + P2
StateGraph(ChatState)  ·  add_node()  ·  set_conditional_entry_point()  ·  compile()
The most critical library in the project. Replaces a simple LangChain chain with a stateful multi-agent graph. Each agent is a node. The router is the conditional entry point. This is what makes the system truly agentic — it routes, branches, and maintains typed state across steps. Without LangGraph this would just be if/else blocks, not an agent system.

Array formatting for FAISS
P1 Only

FAISS requires numpy float32 arrays — cannot work with plain Python lists. numpy converts OpenAI embedding lists into the correct matrix format for FAISS indexing and searching. Minimal direct use but essential dependency. Not needed in Phase 2 since FAISS + BM25 handles its own data format.
Used in: ingestion/vector_store.py + agents/retrieval_agent.py.

Chunk metadata serialisation
P1 Only

FAISS stores only vectors — not the actual text or source filenames. pickle saves a parallel metadata list (chunk_id, text, source filename) to disk alongside the .index file so retrieved vectors can be mapped back to readable content the LLM can use. Simple and fast for local Phase 1 use.
Used in: ingestion/vector_store.py — saves/loads chunk metadata.

HuggingFace models — open-source alternatives and add-ons specific to this RAG project
Context — Why HuggingFace alongside OpenAI
This project uses OpenAI API models (LLaMA 3.1 8B, text-embedding-3-small) as the primary LLM and embedding solution for enterprise KPMG deployment. HuggingFace models serve two purposes here: (1) free local alternatives for development and testing without API costs, and (2) specialised models like cross-encoder re-rankers that HuggingFace excels at and which directly improve RAG quality.
Embedding Model — Alternative to text-embedding-3-small
all-MiniLM-L6-v2
sentence-transformers · Microsoft Research
Phase 1 Dev Alternative
What it does
Converts text chunks and queries into 384-dimensional dense vectors for semantic similarity search. The HuggingFace equivalent of OpenAI's text-embedding-3-small. Runs entirely locally on your laptop CPU — no API key, no cost per call, no internet needed.
Why use this in your project
During local development, calling OpenAI embeddings API for every test run costs money. all-MiniLM-L6-v2 lets you build and test the entire FAISS ingestion pipeline and retrieval agent offline. Same FAISS code works — just swap the embedding function. 384 dims vs 1536 means slightly less precision but 4x faster indexing and free.
from sentence_transformers import SentenceTransformer
model = SentenceTransformer('all-MiniLM-L6-v2')
# Drop-in replacement for OpenAI embeddings:
embeddings = model.encode(chunks, batch_size=32, show_progress_bar=True)
# Update FAISS dim from 1536 to 384
index = faiss.IndexFlatIP(384)

Cross-Encoder Re-Ranker — Improves RAG retrieval quality
ms-marco-MiniLM-L-6-v2
cross-encoder · Microsoft (MS MARCO dataset)
P1 + P2 Optional Add-on
What it does
Re-ranks the top-5 FAISS or AI Search results AFTER initial retrieval. FAISS finds candidates fast using approximate similarity. This model then reads each candidate chunk against the actual query and scores them more precisely — pushing the truly most relevant chunk to position 1.
Why this matters specifically for your PMO RAG
PMO documents have many chunks about similar topics — multiple sections in different reports all mentioning "project risk". Without re-ranking, FAISS might return 5 risk-related chunks but the one specifically answering THIS question could be at position 3. Cross-encoder re-ranking directly improves answer accuracy for KPMG delivery queries.
from sentence_transformers import CrossEncoder
reranker = CrossEncoder('cross-encoder/ms-marco-MiniLM-L-6-v2')
# After FAISS returns top-5:
pairs = [[query, chunk['text']] for chunk in faiss_results]
scores = reranker.predict(pairs)
best_idx = scores.argmax()
top_chunk = faiss_results[best_idx]

Larger Embedding Model — Higher precision for complex docs
all-mpnet-base-v2
sentence-transformers · Microsoft
Phase 1 Upgrade Option
What it does
Higher quality embedding model than MiniLM. Produces 768-dimensional vectors with better semantic understanding. Better for dense, complex PMO language — delivery frameworks, risk taxonomies, project management jargon — where MiniLM might miss subtle semantic distinctions.
When to choose this over MiniLM
When retrieval accuracy matters more than speed. For KPMG SOPs and delivery reports with complex technical language, mpnet-base captures meaning more accurately. Trade-off: 5x more parameters, 2x slower than MiniLM, but still free and fully local.
from sentence_transformers import SentenceTransformer
model = SentenceTransformer('all-mpnet-base-v2')
embeddings = model.encode(chunks)
# Update FAISS from 384 to 768 dimensions:
index = faiss.IndexFlatIP(768)

Local LLM — Full offline alternative to LLaMA 3.1 8B
Phi-3-mini-4k-instruct
Microsoft · HuggingFace Hub
Phase 1 Local Dev Only
What it does
Small but capable instruction-tuned LLM from Microsoft. Can replace LLaMA 3.1 8B for local testing of all three agents — document Q&A in the Retrieval Agent, KPI formatting in the API Agent, and workflow action classification in the Workflow Agent. Runs on a laptop CPU without any API key.
Why relevant to your project specifically
During development, calling LLaMA 3.1 8B for every agent test costs money and requires internet. Phi-3-mini lets you test all LangGraph routing logic, agent prompts, and JSON extraction patterns completely offline. When satisfied, swap back the client to LLaMA 3.1 8B for production quality. Same prompt format — easy swap.
from transformers import pipeline
pipe = pipeline("text-generation",

  trust_remote_code=True)
# Same prompt format as LLaMA 3.1 8B:
response = pipe(prompt, max_new_tokens=500)[0]['generated_text']
