🔤 Foundation Concepts

# Concept Definition Used In Project Interview Answer
1 LLM (Large Language Model) Neural network trained on billions of text tokens for language understanding and generation Groq LLaMA 3.1 in ALL agents We use LLaMA 3.1 8B via Groq API for routing, extraction, parsing, and generation
2 Transformer Architecture using self-attention for parallel text processing LLaMA 3.1 (decoder) + MiniLM (encoder) LLaMA is decoder-only, MiniLM is encoder-only — both transformer-based
3 Attention Mechanism Allows model to focus on relevant input parts during generation Inside both LLaMA and MiniLM Self-attention is why transformers understand context better than RNNs
4 Inference Running trained model for predictions (not training) Every Groq API call We only do inference — no training. Groq uses custom LPU hardware
5 Token Smallest unit of text for LLM processing (~4 chars = 1 token) LLaMA 3.1 = 128K token context We send ~1000 tokens of context per query, well within 128K limit

🧠 Embeddings & Vector Search

# Concept Definition Used In Project Interview Answer
6 Embedding Converting text into fixed-size numerical vector capturing semantic meaning ingestion/embedder.py — 384-dim vectors We use all-MiniLM-L6-v2 for 384-dim embeddings; similar texts → similar vectors
7 Sentence Transformer Transformer fine-tuned for sentence-level embeddings all-MiniLM-L6-v2 from HuggingFace Trained on 1B+ sentence pairs with contrastive learning
8 Vector Database / Store DB optimized for similarity search on vectors FAISS IndexFlatL2 We use FAISS — a library, not a cloud DB. Perfect for <100K vectors
9 Cosine Similarity Measures angle between vectors (1.0 = identical) Router Tier 2 Magnitude-invariant — 5-word query matches 20-word anchor if same meaning
10 L2 Distance (Euclidean) Straight-line distance between vectors FAISS IndexFlatL2 For normalized vectors, L2 and cosine give equivalent rankings
11 Nearest Neighbor Search Finding K closest vectors to query FAISS top_k=5 Exact NN search — brute force is faster than HNSW for <100K vectors
12 Dimensionality Number of values per vector DIMENSION = 384 384 dims balances accuracy vs speed. OpenAI uses 1536 — 4x slower

📚 RAG (Retrieval-Augmented Generation)

# Concept Definition Used In Project Interview Answer
13 RAG Retrieve relevant docs, then generate answer grounded in those docs Retrieval Agent core pattern We retrieve top-5 chunks from FAISS, inject as context, LLM answers from context only
14 Chunking Splitting docs into smaller pieces for LLM context 500-char chunks, 100-char overlap Overlap prevents cutting mid-sentence. Each chunk = one FAISS vector
15 Context Window Max text LLM can process at once LLaMA 3.1 = 128K tokens We use ~1000 tokens of context — well within 128K limit
16 Grounded Generation LLM answers ONLY from provided context System prompt enforcement Anti-hallucination: must cite documents, say "I don't know" if not in context
17 Semantic Search Search by meaning, not exact words FAISS search + Router Tier 2 "financial burn" finds "budget expenditure" because vectors are similar
18 Document Ingestion Pipeline: Parse → Chunk → Embed → Store ingestion/ pipeline Runs once: parse → 500-char chunks → MiniLM embed → FAISS. ~10s for 50 docs

🤖 Agent Architecture

# Concept Definition Used In Project Interview Answer
19 Multi-Agent System Multiple specialized agents, each expert in one domain 4 agents: Retrieval, API, Helpdesk, Workflow Each agent has own tools, prompts, data sources — independently testable
20 Agent Orchestration Coordinating which agent handles which request LangGraph StateGraph LangGraph uses conditional edges to route to the right node
21 State Machine Typed state dict flowing through nodes/edges ChatState TypedDict State carries query, session_id, answer, agent_used through the graph
22 Conditional Routing Different paths based on classification route_query() → 4 branches 3-tier routing: keyword → embedding → LLM fallback
23 Tool Use Agent calling external APIs/functions Connectors = agent tools Each agent has specialized connectors (SQL, Jira, ServiceNow, etc.)

🎯 LLM Techniques

# Concept Definition Used In Project Interview Answer
24 Prompt Engineering Crafting instructions for LLM behavior System prompts per agent Each agent has specialized system prompts tuned for its task
25 System Prompt Instructions that set LLM behavior/persona "Answer ONLY from context" Grounding prompt prevents hallucination in retrieval agent
26 Temperature Controls randomness (0=deterministic, 1=creative) 0 / 0.2 / 0.3 per task 0 for extraction, 0.2 for factual, 0.3 for formatting
27 Few-Shot Learning Examples in prompt to guide output JSON extraction examples We include 2-3 examples in prompts for consistent JSON output
28 Zero-Shot Classification LLM classifies without examples Router Tier 3 Fallback when keyword + embedding fail — LLM picks the right agent
29 JSON Mode Forcing LLM to output valid JSON Helpdesk/Workflow extraction We parse action, fields, urgency from natural language into structured JSON
30 Chain-of-Thought Step-by-step reasoning in prompts Complex metric explanations Helps LLM break down multi-step questions about KPIs and project health

🤗 HuggingFace & Model Details

# Concept Definition Used In Project Interview Answer
42 HuggingFace Hub Repository of pre-trained ML models all-MiniLM-L6-v2 download ~80MB model downloaded once, cached locally for fast inference
43 Model Card Documentation for model capabilities MiniLM: 22M params, 384 dims 22M params makes it lightweight — loads in ~2s, runs on CPU
44 Fine-Tuning Training pre-trained model on specific data MiniLM fine-tuned on 1B+ pairs We don't fine-tune — we use the pre-trained model as-is
45 Transfer Learning Using knowledge from one task for another MiniLM trained on NLI → search Model trained on NLI task transfers well to semantic search
46 Contrastive Learning Training by comparing similar/dissimilar pairs How MiniLM learned embeddings Similar sentences → close vectors, dissimilar → far vectors
